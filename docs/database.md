# Database Architecture

The CrowdCanvas database is built on **Supabase (PostgreSQL 15+)**. It is a single, unified schema designed for strict security, high relational integrity, and advanced AI vector operations.

## Core Schema Overview

The database is divided into logical domains:

1.  **Identity & Access:** `profiles`, `user_sessions`
2.  **Core Event Logic:** `events`, `event_members`, `event_role_requests`
3.  **Media & Content:** `media`
4.  **Social Interactions:** `likes`, `comments`, `shares`, `media_favourites`, `user_pinned_events`
5.  **Notifications:** `notifications`
6.  **Face Recognition:** `face_profiles`, `media_faces`, `face_matches`, `photo_user_tags`, `user_privacy_settings`
7.  **Personalization (AI):** `user_preference_profiles`, `user_behavior_events`, `recommendation_cache`, `recommendation_analytics`

---

## Custom Types (Enums)

To enforce strict data integrity at the database layer, we use Postgres Enums:

*   `user_role`: ('admin', 'photographer', 'member', 'viewer')
*   `event_member_role`: ('owner', 'admin', 'uploader', 'viewer')
*   `share_type_enum`: ('copy_link', 'whatsapp', 'twitter', 'facebook', 'download')
*   `tag_status`: ('pending', 'approved', 'rejected', 'removed')
*   `notification_type`: Defines specific trigger events for the notification system.

---

## Vector Operations (`pgvector`)

CrowdCanvas uses the `pgvector` extension for two distinct similarity searches:

1.  **Semantic Media Search:** The `media` table contains an `embedding VECTOR(768)` column. This stores sentence-transformers embeddings of the AI-generated image descriptions.
2.  **Facial Recognition:** The `face_profiles` and `media_faces` tables contain an `embedding VECTOR(128)` column. This stores the facial descriptors generated by `face-api.js`.

### RPC Functions (Remote Procedure Calls)

We execute vector math natively in Postgres using RPCs for massive performance gains:

*   `match_media`: Finds media similar to a search query embedding using cosine distance (`<=>`).
*   `match_faces`: Compares an unknown face against the database of known faces to suggest tags.

---

## Database Triggers

Triggers are used to maintain referential integrity without application-level logic:

*   **`on_auth_user_created`**: When a user signs up via Supabase Auth, this trigger automatically creates a row in the public `profiles` table.
*   **`on_event_created`**: When an event is created, this trigger automatically assigns the creator as the `owner` in the `event_members` table.
*   **`trg_enforce_notification_retention`**: Automatically deletes old notifications when a user exceeds 25 notifications to save space.
